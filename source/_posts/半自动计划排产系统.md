## 前言

在制造业干过的人都知道，排产这活儿，说简单也简单——不就是把订单分配到机台上嘛。但真做起来，你会发现这玩意儿的复杂度是指数级增长的。

我所在的公司是做线缆的，产品从绝缘、对绞、成缆、编织、护套到成圈，六道工序串下来，每道工序都有不同的设备、不同的速度、不同的调机时间。一个订单可能涉及几十种原材料，有些物料要提前半个月采购，齐套了才能开始生产。

**以前是怎么排产的？**

Excel:

![旧的排产Excel表格](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/Excel%E6%8E%92%E4%BA%A7%E8%A1%A8%E6%A0%BC.png)

计划员每天早上打开Excel，看看订单列表，凭经验判断哪个订单先做、分配到哪台机器、需要多少时间。遇到紧急订单就打乱重排，遇到物料没到就干等。整套流程下来，至少要花2-3个小时，而且经常出错——排重了、漏排了、算错产能了，各种问题层出不穷。

我决定用技术来解决这个问题。

---

## 一、先搞清楚业务到底要什么

在写第一行代码之前，我花了整整一周时间跟计划员、车间主任、采购员聊天。把他们的痛点整理出来，最后归纳成14条核心需求：

![白板需求梳理](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/%E7%99%BD%E6%9D%BF%E9%9C%80%E6%B1%82%E6%A2%B3%E7%90%86.jpg)

### 痛点清单

| # | 业务痛点 | 现状 |
|---|---------|------|
| 1 | 排产全靠人工，2-3小时/天 | Excel手动排 |
| 2 | 订单交期倒排算不清楚 | 凭经验估算 |
| 3 | 物料齐套状态不透明 | 打电话问仓库 |
| 4 | 设备产能没有数据支撑 | 拍脑袋分配 |
| 5 | 复杂订单容易漏排 | 没有提醒机制 |
| 6 | 排产后无法追踪执行 | MES数据对不上 |
| 7 | 采购建议靠人工计算 | 经常多买或少买 |

这些问题看起来独立，但实际上是一个环环相扣的链路：**订单 → 工艺结构 → 物料需求 → 齐套判断 → 产能分配 → 排产下发 → 执行追踪**。

我要做的不是一个简单的排产表格，而是一套完整的计划管理系统。

## 二、系统架构设计

技术选型上，还是我最熟悉的组合：

- **后端**: Django + Django REST Framework + MySQL
- **前端**: Vue3 + Element Plus + TypeScript
- **任务调度**: Celery + Redis
- **数据集成**: 对接 ERP（SQL Server）和 MES 系统

整体数据流是这样的：

![整体数据流转](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/%E6%8E%92%E4%BA%A7%E6%A0%B8%E5%BF%83%E6%9E%B6%E6%9E%84%E5%9B%BE.png)

### 核心数据模型

整个系统围绕几张核心表展开：

```
PlanTotal (排产计划总表)
    ├── ProductProcessStructure (产品工艺结构)
    ├── ProcessWorkTime (工序标准工时)
    ├── ProcessWorkMachine (工序设备产能)
    │       └── MachineDefinition (设备定义)
    │               └── VirtualMachine (虚拟机台/线速度)
    ├── MaterialInventory (物料库存)
    ├── ProcessMaterialRequirement (工序齐套需求)
    └── ProcessWorkPlan (排产记录)
```

这里有个设计决策值得说一下：**虚拟机台**的概念，这是计划的老大提出来的。

最初线速度是直接绑定在设备上的，但实际生产中，同类型的设备（比如都是绝缘机）线速度是一样的。如果每台设备都配一次线速度，维护成本太高。所以我抽象了一层"虚拟机台"——定义线速度类别，设备关联虚拟机台，工序标准工时也关联虚拟机台。一次配置，多处复用。

```
虚拟机台（绝缘机, 线速度150m/h）
    ├── 设备A（一车间, 绝缘机1号）
    ├── 设备B（一车间, 绝缘机2号）
    └── 工序标准工时（绝缘 × 产品P-CAT6 = 150m/h + 调机0.5h）
```

## 三、核心功能详解

### 3.1 交期倒排——从交期反推每个工序的最晚开工时间

这是整个系统的灵魂。原理不复杂，但实现起来细节很多。

**公式：**
```
订单净耗时 = Σ(订单数量 / 线速度 × 倍数因子 × 余量系数) 对每道工序求和
```

其中"倍数因子"是根据产品特性来的：
- 绝缘工序：芯数（比如8芯的线，绝缘要做8次）
- 对绞工序：对绞数
- 成缆/编织/护套：各自的倍数

举个例子：一个6类网络线订单，数量12000米，芯数8，对绞数4：

| 工序 | 线速度 | 倍数 | 余量系数 | 工时(小时) |
|------|--------|------|---------|-----------|
| 绝缘 | 150m/h | ×8 | ×1.05 | 672 |
| 对绞 | 150m/h | ×4 | ×1.03 | 329.6 |
| 成缆 | 80m/h | ×1 | ×1.0 | 150 |
| 编织 | 150m/h | ×1 | ×1.0 | 80 |
| 护套 | 90m/h | ×1 | ×1.02 | 136 |
| 成圈 | 200m/h | ×1 | ×1.0 | 1.5 |
| **合计** | | | | **1369.1** |

从交期往前倒推，就能算出每道工序的最晚开工时间，进而推算出原材料的最晚到货时间。如下图: 齐套看板

### 3.2 物料齐套判断——不再打电话问仓库

倒排计算完成后，系统会自动对比每个工序的物料需求和当前库存：

```python
需求量 = 订单数量 × BOM用量
缺口量 = max(需求量 - 当前库存, 0)
是否齐套 = (缺口量 == 0)
```

如果物料不够，系统会生成采购建议，包含建议采购量（考虑最小起订量）和最晚到货时间。

![齐套看板](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/%E4%BA%A4%E6%9C%9F%E5%80%92%E6%8E%92%E7%BB%93%E6%9E%9C%E9%A1%B5%E9%9D%A2.png)

### 3.3 半自动排产——人机协同

为什么叫"半自动"？因为全自动排产在实际生产中几乎不可能实现——要考虑的因素太多：设备状态、工人排班、紧急插单、物料到货延迟……

我的策略是：**系统算，人来拍**。

1. **系统负责**：计算每个订单每个工序的可排数量、齐套时间、产能负荷
2. **人负责**：决定哪个订单先排、排多少、排到哪台机器

具体流程：

![排产看板](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/%E6%8E%92%E4%BA%A7%E7%9C%8B%E6%9D%BF.png)

**排产操作：**
1. 从订单池拖拽订单到目标机台+日期
2. 弹窗显示该订单各工序的剩余可排量
3. 输入本次排产数量（支持部分排产）
4. 系统校验数量不超过剩余量，确认后完成排产

**部分排产**是个关键功能。现实中很少有订单能一次性全部排下去——可能某台机器产能不够，可能物料只到了一部分。所以一个订单可以分多次排产，系统自动跟踪每个工序的已排/剩余数量。

![排产数量弹窗](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/%E6%8E%92%E4%BA%A7%E7%9C%8B%E6%9D%BF-%E5%BC%B9%E7%AA%97.png)


### 3.4 产能负荷度——一眼看出哪台机器要爆

排产看板上每个机台格子顶部有一个负荷度指示条：

```
负荷度 = 当天该机台排产总量 / 机台标准产能 × 100%
```

颜色编码：
- 🟢 < 50%：低负荷，还有很多余量
- 🔵 50%-80%：正常
- 🟠 80%-100%：高负荷，快满了
- 🔴 ≥ 100%：超负荷，需要调整

![负荷度显示](https://fenixtc-1258871859.cos.ap-guangzhou.myqcloud.com/%E8%AE%A1%E5%88%92%E6%8E%92%E4%BA%A7/%E8%B4%9F%E8%8D%B7%E5%BA%A6%E6%98%BE%E7%A4%BA.png)


### 3.5 设备状态联动——停机的机器不会被排

设备状态从MES实时同步，只有"可用"状态的设备才会被排产系统选中。停机或故障的设备在看板上显示灰色，提醒计划员换机器。


### 3.6 复杂订单自动标记

有些订单因为芯数多、数量大、工艺特殊，属于"复杂订单"，需要额外关注。系统支持配置判断规则：

- 芯数 ≥ 8 → 复杂
- 订单数量 > 9000 → 复杂

导入订单后自动打标，在订单池中一目了然。

### 3.7 ERP/MES数据自动同步

手动录入数据太容易出错，所以我写了两个同步脚本：

**ERP订单同步**（每天凌晨2点，Celery定时）：
```python
# 从ERP拉取活跃订单，按指令号去重写入计划总表
python manage.py sync_erp_orders
```

**MES设备同步**（手动执行）：
```python
# 从MES拉取设备数据，自动匹配工序、提取车间信息
python manage.py sync_mes_devices
```

ERP的SQL查询比较复杂，用了游标+子查询来计算每个订单的芯数、对绞数等工序倍数因子。这些数据在ERP里是分散在多个表里的，我做了一次性聚合。

## 四、技术实现中的几个坑

### 4.1 排产数量不能跨工序简单相加

最初设计的时候，我把排产数量简单理解为"订单总量"。但实际上，每个工序的数量是不一样的——绝缘工序的数量 = 订单数量 × 芯数。如果一个8芯12000米的订单，绝缘工序要处理的是 12000 × 8 = 96000 米。

这个坑导致排产后剩余量计算错误，订单池里的数量对不上。后来我把剩余量的计算改成了按工序维度：

```python
# 错误：按订单总量计算
remaining = bill_amount - sum(all_process_meters)

# 正确：按工序分别计算
remaining_per_process = process_total - scheduled_for_this_process
```

### 4.2 同一订单同一机台要合并

用户拖拽同一个订单到同一台机器的同一日期，应该合并数量而不是创建两条记录。这个在后端做了合并逻辑：

```python
existing = ProcessWorkPlan.objects.filter(
    ZLH=plan.ZLH,
    MachineNo=machine.MachineNo,
    PlanTime__date=schedule_day,
    scheduleStatus__in=["0", "1"],
).first()

if existing:
    existing.Meters += plan_amount
    existing.save()
else:
    ProcessWorkPlan.objects.create(...)
```

### 4.3 齐套时间过期的订单不能自动排产

有些订单的齐套时间已经过了（比如物料应该上周到但还没到），这种订单如果自动排产，排出来的时间全是过去的，没有意义。所以加了一个判断：齐套时间 < 今天 → 只入池不排产，等物料到了再手动排。

### 4.4 Sortable.js拖拽的坑

前端用 Sortable.js 实现拖拽排产。最初配置了 `pull: true`，结果拖走订单后，订单池里的原卡片就没了——因为 `pull: true` 是移动模式，不是克隆模式。改成 `pull: 'clone'` 才解决。

还有一个问题是拖拽后数据更新了但DOM没刷新，最后用 `loadKey` 强制重建组件才搞定。

## 五、第三方接口设计

系统需要对接第三方（MES、ERP等），我设计了三个标准数据同步接口：

| 接口 | 用途 | 频率 |
|------|------|------|
| MES报工同步 | 实时接收生产报工数据 | 每次报工触发 |
| 库存同步 | 更新原材料/半成品库存 | 每小时或变动时 |
| 设备状态同步 | 更新设备可用状态 | 状态变更时 |

所有接口支持批量推送，统一返回格式，配套完整的接口文档。

## 七、后续规划

系统还有很多可以优化的地方：

1. **智能排产算法**：基于遗传算法或模拟退火，自动给出最优排产方案，人工只需微调
2. **APS高级排程**：考虑更多约束条件（工人技能、模具、换线时间等）
3. **移动端**：车间主任用手机就能查看排产和报工
4. **数据大屏**：实时展示产能利用率、订单进度、设备状态

## 写在最后

做这个项目最大的感触是：**技术不是最难的，理解业务才是**。

排产系统的逻辑本身不复杂——加减乘除而已。但要把业务场景的方方面面都考虑到，需要大量的沟通和试错。一个公式算错了，可能导致整个排产结果不可用；一个边界条件没考虑到，可能导致系统在生产环境崩溃。

从Excel到系统，改变的不只是工具，而是整个工作方式。计划员从"凭经验排"变成了"看数据排"，采购从"打电话问"变成了"系统自动算"，车间从"等通知"变成了"看板实时看"。

这就是技术的价值——让复杂的事情变简单，让不确定的事情变确定。
